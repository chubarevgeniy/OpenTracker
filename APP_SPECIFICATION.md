# Спецификация приложения "Calorie Tracker"

Этот документ содержит полное техническое описание приложения "Calorie Tracker". Цель документа — предоставить исчерпывающую информацию для создания клона или аналогичного приложения с тем же функционалом, стилем и поведением.

## 1. Общее описание

"Calorie Tracker" — это мобильно-ориентированное веб-приложение (PWA-подобное, Offline-First) для трекинга калорий, БЖУ (белки, жиры, углеводы) и веса. Приложение позволяет искать продукты по базе (или сканировать штрихкоды), логировать приемы пищи, отслеживать статистику, а также импортировать/экспортировать данные. Все данные сохраняются локально на устройстве пользователя.

## 2. Стек технологий

*   **Фреймворк:** React 19 (c TypeScript)
*   **Сборщик:** Vite
*   **Стилизация:** Tailwind CSS v4 (используется `@theme` в `index.css`, без отдельного `tailwind.config.js`). Дополнительно `clsx` и `tailwind-merge` для удобного объединения классов.
*   **Роутинг:** `react-router-dom` (используется `HashRouter` для совместимости с GitHub Pages).
*   **Управление состоянием (State Management):** Zustand.
*   **Локальное хранилище данных (Persistent Storage):** IndexedDB (через библиотеку `localforage`, интегрированную с Zustand Persist middleware). `idb-keyval` также может использоваться как альтернатива или дополнение.
*   **HTTP-клиент:** Axios (с кастомной логикой повторных попыток запроса вместо react-query).
*   **Иконки:** `lucide-react`.
*   **Графики (Data Visualization):** `recharts`.
*   **Работа с датами:** `date-fns`.
*   **Сканирование штрихкодов:** `html5-qrcode`.
*   **Распознавание текста (OCR) для импорта с изображений/скриншотов:** `tesseract.js` (работает локально на клиенте).

## 3. Архитектура и структура проекта

Все команды (`pnpm install`, `pnpm dev`, `pnpm build`, `pnpm lint`) должны выполняться в директории `calorie-tracker`.

### Структура `src/`

*   `assets/` - статические файлы, иконки.
*   `components/` - переиспользуемые UI-компоненты (например, `Layout` с нижней навигацией, `MealEntry`, `Scanner`).
*   `lib/` - утилиты общего назначения (например, `utils.ts` для объединения Tailwind классов).
*   `pages/` - компоненты-страницы, привязанные к роутам (`Dashboard`, `Search`, `Settings`, `Stats`).
*   `services/` - взаимодействие с внешними API (например, `api.ts` для Open Food Facts).
*   `store.ts` - глобальное состояние приложения на базе Zustand (включая логику дневника, настроек пользователя и кастомной еды).
*   `utils/` - бизнес-логика и хелперы (`calculations.ts`, `export.ts`, `import.ts`, `imageParser.ts`).

Также в корне проекта находится папка `agents/`, содержащая markdown-документы (например, `CSV Compatibility Agent`) с правилами и инструкциями для ИИ-агентов, определяющими логику поддержки обратной совместимости, особенно при импорте/экспорте.

### Особенности развертывания

*   Деплой производится на GitHub Pages (например, `chubarevgeniy.github.io/OpenTracker/`).
*   Параметр `base` в `vite.config.ts` и `basename` в роутере должны быть настроены на `/OpenTracker/`.

## 4. UI / UX дизайн и стилизация

Приложение имеет строго мобильную направленность и специфический узнаваемый визуальный стиль.

### 4.1. Основные визуальные характеристики
*   **Закругления:** Интенсивное использование закругленных углов на карточках, кнопках и полях (например, Tailwind классы `rounded-2xl`, `rounded-3xl`, `rounded-full`).
*   **Акцентный цвет:** Неоновый зеленый (`#c5f82a` / переменные `--theme-primary`).
*   **Карточки (Cards):** Карточки не имеют жестких границ (`border: none`). Вместо этого используются мягкие кастомные тени: `shadow-[0_4px_20px_rgba(0,0,0,0.03)]`.
*   **Отсутствие outline:** Глобальные рамки фокуса (focus outlines) и рамки у оберток графиков Recharts намеренно отключены (`outline: none !important` в `index.css`) в угоду визуальной чистоте. Вместо стандартного `outline` используется визуальная обратная связь при фокусе (например, изменение `bg` или `ring`).

### 4.2. Темы (Light / Dark Mode)
*   Выбор темы (light, dark, system) управляется через хранилище Zustand (`settings.theme`).
*   Применение темы происходит путем переключения класса `dark` на корневом HTML-элементе.
*   В Tailwind используются CSS-переменные для семантических цветов (например, `bg-surface`, `text-text`), которые определены в `@theme` в `index.css`.
*   Там, где требуется переопределение или смягчение цветов в темной теме (например, для опасных кнопок - danger buttons), используются явные классы `dark:` с пониженной непрозрачностью (например, `dark:bg-red-500/10`, `dark:text-red-400`).

### 4.3. Адаптивность и Layout
*   **Mobile-first:** Приложение разрабатывалось в первую очередь для смартфонов, имеет нижнюю панель навигации (Bottom Navigation).
*   **Viewport Height:** Для избежания проблем с прыжками интерфейса из-за скрывающихся адресных строк в мобильных браузерах используется `100dvh` (вместо стандартного `100vh` или `min-h-screen`). Обертка имеет `overflow-hidden`, а внутренний контейнер для контента — `overflow-y-auto`.

### 4.4. Доступность (Accessibility / a11y)
*   На всех кнопках, состоящих только из иконок, обязательно присутствуют `aria-label`.
*   Поддерживается управление с клавиатуры для основных элементов.
*   Стилизация выпадающих списков (`<select>`): используется класс `appearance-none` (плюс отступ справа, например `pr-10`) внутри относительной обертки (`relative`), в сочетании с кастомной SVG-иконкой стрелки (с абсолютным позиционированием), чтобы избежать наложения системных стрелок.

## 5. Работа с данными и состояние (State)

### 5.1. Структура `dailyLogs`
Основная структура данных для хранения дневника питания (`dailyLogs` в Zustand) — это словарь (Record), ключом в котором является строковая дата в формате `yyyy-MM-dd`: `Record<string, DailyLog>`.
*Важно:* Для максимальной производительности (особенно при рендерах) следует использовать прямое обращение по ключу (поиск по дате) вместо `Object.values(dailyLogs).filter/sort`.

### 5.2. Уникальные идентификаторы
Для всех локальных сущностей (пользовательская еда, разобранные элементы) генерируются уникальные идентификаторы с помощью `crypto.randomUUID()` (UUID v4), чтобы избежать коллизий ID с внешними источниками.

### 5.3. Импорт и Экспорт данных (CSV)
*   **Безопасность:** Данные из внешних источников строго валидируются и санируются (удаляются HTML-теги) перед мержем в стейт, чтобы предотвратить XSS.
*   **Shallow Merge:** Действие `importData` в Zustand выполняет поверхностное слияние (shallow merge) в `dailyLogs`. Чтобы при частичном импорте CSV (например, только веса) не удалились уже существующие данные о приемах пищи за этот день, утилита `importFromCSV` сохраняет существующие поля для невыбранных опций.
*   **Формат файлов:** В input для CSV (`type="file"`) обязательно указывается широкий набор MIME-типов (`accept=".csv,text/csv,application/vnd.ms-excel"`) для обеспечения совместимости нативного file picker'а в iOS, Android и десктопных ОС.
*   **Обратная совместимость (Смотри `agents/`):** Столбец `Date` в CSV всегда должен быть первым. Структурные изменения (например, экспорт сложных структур) добавляются в конец строки, чтобы не сломать парсер.

### 5.4. Распознавание изображений (OCR)
*   Приложение поддерживает импорт данных (макронутриентов) со скриншотов через библиотеку `tesseract.js`.
*   Поддерживается массовый (bulk) импорт — можно загрузить сразу несколько скриншотов в разделе настроек.
*   В `imageParser.ts` используется "нежадный" (non-greedy) поиск по регулярным выражениям (`.*?`), чтобы извлекать данные о белках, жирах и углеводах, так как OCR часто дает артефакты и ошибки в единицах измерения.

### 5.5. Графики (Recharts)
*   Компоненты библиотеки `recharts` (например, `<Tooltip>`) для корректной поддержки темной темы требуют явного маппинга CSS-переменных темы (например, `var(--theme-surface)`, `var(--theme-text)`) внутри атрибута `style` (inline styles).

## 6. Внешние API и сервисы
*   **Open Food Facts API:** Используется для поиска продуктов по названию и по штрихкоду.
*   Запросы выполняются через Axios.

## 7. Функциональные возможности

1.  **Дневник (Dashboard):** Логирование Завтрака, Обеда, Ужина и Перекусов. Подсчет выпитой воды.
2.  **Поиск и Сканер (Search):** Поиск по базе Open Food Facts и локальной пользовательской базе. Сканирование штрихкодов через камеру устройства.
3.  **Персональные цели:** Расчет суточной нормы калорий (BMR) и макронутриентов по формуле Миффлина-Сан Жеора, исходя из параметров пользователя (пол, возраст, рост, вес, активность).
4.  **Статистика (Stats):** Графики потребления калорий и изменения веса (Recharts).
5.  **Расчет реального TDEE:** Приложение умеет вычислять реальный ежедневный расход энергии (TDEE), если пользователь стабильно вносит данные о еде и весе в течение 14+ дней.
6.  **Настройки и импорт/экспорт:** Изменение параметров тела, смена темы оформления, управление базой кастомных продуктов, импорт/экспорт всей истории в CSV, массовый импорт со скриншотов с использованием локального OCR.

## 8. Правила для разработчика (Constraint summary)
*   Использовать только `pnpm`.
*   Не добавлять новые тяжелые UI-библиотеки без согласования.
*   Соблюдать текущий дизайн: округлые формы, безбордерные карточки с тенями, неоново-зеленый акцентный цвет.
*   Не переписывать backend/performance логику, если это не требуется явно.
*   Соблюдать правила из `agents/*.md`.
*   Следить за чистотой репозитория от временных файлов.
